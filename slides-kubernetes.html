Slide 02 — Introducción

¿Qué es Kubernetes?

🚢

Orquestador de contenedores

Kubernetes automatiza el despliegue, escalado y operación de aplicaciones en contenedores. El nombre viene del griego "timonel de barco".

🔧

Open Source desde 2014

Creado por Google y donado a la CNCF. Basado en el sistema interno Borg. Actualmente el estándar de la industria para orquestación.

⚡

Auto-recuperación

Reinicia contenedores fallidos, reemplaza pods caídos, elimina los que no pasan health checks y no los expone hasta que estén listos.

📦

Agnóstico a la nube

Funciona en GCP (GKE), AWS (EKS), Azure (AKS), on-premise o incluso en tu laptop con minikube/kind.

En este lab usamos Google Kubernetes Engine (GKE) con namespace gcplab

🧠 Control Plane

API Server: punto de entrada de todas las operaciones.
etcd: base de datos distribuida del estado del clúster.
Scheduler: asigna pods a nodos.

🖥️ Worker Node

kubelet: agente que ejecuta los pods.
kube-proxy: maneja el networking.
Container Runtime: Docker / containerd.

📦 Pod

Unidad mínima de K8s. Uno o más contenedores que comparten red y almacenamiento. En este lab: gcp-lab-app en puerto 3000.

Slide 04 — Namespaces

Aislamiento con Namespaces

# devops/01-namespace.yaml apiVersion: v1 kind: Namespace metadata: name: gcplab labels: app: gcp-labs creator: jeschb course: gcp-fundamentos-2026 annotations: description: "Namespace para Laboratorio de Kubernetes en GCP"

🔲 ¿Para qué sirven?

Particionan un clúster físico en múltiples clústeres virtuales. Permiten separar ambientes: dev, staging, prod.

🏷️ Labels y Annotations

Labels: pares clave-valor para selección y agrupación.
Annotations: metadata arbitraria legible por humanos o herramientas.

🛡️ Aislamiento de recursos

Con ResourceQuota y LimitRange puedes controlar cuántos recursos consume cada namespace.

kubectl get all -n gcplab → lista todos los recursos del namespace
Slide 05 — Deployments

Deployments y ReplicaSets

📋 Deployment gcp-lab-app

Describe el estado deseado. Kubernetes trabaja continuamente para mantenerlo.
Imagen: gcr.io/[PROJECT_ID]/gcp-labs-kubernetes:latest

🔄 Rolling Update (Zero Downtime)

maxSurge: 1 — puede haber 1 pod extra durante la actualización.
maxUnavailable: 0 — nunca baja la disponibilidad durante el rollout.

📐 Escalado horizontal

Empezamos con 1 réplica. Se puede escalar en caliente:

kubectl scale deployment gcp-lab-app --replicas=3 -n gcplab
# devops/02-deployment.yaml (extracto) kind: Deployment spec: replicas: 1 strategy: type: RollingUpdate rollingUpdate: maxSurge: 1 maxUnavailable: 0 template: spec: containers: - name: gcp-lab-app image: gcr.io/.../gcp-labs-kubernetes ports: - containerPort: 3000

ConfigMap & Secret

📋 ConfigMap — gcplab-cm

Almacena configuración no sensible como variables de entorno o archivos de configuración.

🔐 Secret — gcplab-secret

Almacena datos sensibles en Base64. ⚠️ No es cifrado — para producción usa Sealed Secrets o GCP Secret Manager.

# devops/06-secret.yaml type: Opaque data: # valores en Base64 (no cifrado) APP_CREATION_TIMESTAMP: MjAyNi0wNC0wNlQwMD... TEACHER_EMAIL: amVzY2hiQGdtYWlsLmNvbQ== TEACHER_NAME: SW5nIEplc8O6cy4uLg==
⚠️ Nunca subas Secrets sin cifrar a git. Usa .gitignore o herramientas como Sealed Secrets / SOPS.
Slide 08 — Observabilidad

Health Checks: Liveness & Readiness

Liveness Probe

💓 ¿Está vivo el pod?

Si falla → K8s reinicia el contenedor. Detecta deadlocks o estados irrecuperables.

livenessProbe: httpGet: path: /api/pod-info port: 3000 initialDelaySeconds: 20 periodSeconds: 10 timeoutSeconds: 5
Readiness Probe

✅ ¿Listo para tráfico?

Si falla → K8s lo saca del Service (no le manda peticiones). Útil durante warmup o carga de datos.

readinessProbe: httpGet: path: /api/pod-info port: 3000 initialDelaySeconds: 10 periodSeconds: 5 timeoutSeconds: 3
Pod Disruption Budget

🛡️ Alta Disponibilidad

Garantiza que siempre haya mínimo 1 pod disponible incluso durante mantenimiento del nodo o actualizaciones del clúster.

kind: PodDisruptionBudget spec: minAvailable: 1 selector: matchLabels: app: gcp-lab-app
Slide 09 — Recursos & Variables

Gestión de Recursos e Inyección de Variables

🖥️ CPU

1000m = 1 núcleo virtual

Request: 100m (mínimo garantizado)
Limit: 500m (máximo permitido)

💾 Memoria

Mi = Mebibyte (1 Mi ≈ 1.05 MB)

Request: 128Mi (mínimo garantizado)
Limit: 512Mi (máximo permitido)

🌐 Variables de entorno del Pod

K8s inyecta metadata del pod en tiempo de ejecución vía Downward API:

env: - name: POD_NAME valueFrom: fieldRef: fieldPath: metadata.name - name: POD_IP valueFrom: fieldRef: fieldPath: status.podIP
Por qué importa: Los requests determinan en qué nodo se programa el pod; los limits evitan que un pod consuma todos los recursos del nodo y afecte a sus vecinos.
Slide 10 — Flujo de Despliegue

Flujo de Despliegue en GCP + kubectl

  • 1

    Configurar GCP

    gcloud auth configure-docker
    Autenticar Docker con Google Container Registry.

  • 2

    Build imagen Docker

    docker build -t gcr.io/[PROJECT]/gcp-labs-kubernetes .

  • 3

    Push a GCR

    docker push gcr.io/[PROJECT]/gcp-labs-kubernetes

  • 4

    Aplicar manifests en orden

    kubectl apply -f devops/
    namespace → configmap → secret → deployment → service

  • 5

    Acceder a la app

    kubectl port-forward pod/[NAME] 3000:3000 -n gcplab
